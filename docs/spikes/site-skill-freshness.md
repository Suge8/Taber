# 站点技能新鲜度反馈 spike

日期：2026-07-14
基线：`7907907..e19d634` 及当前工作区

漂移检查只发现 `lib/agent-tools.ts` 的先序审计闸门和 target 串行化改动。技能状态仍只有 `Skill` 行、任务内 `StaleSkillTracker` 和一次性失效提示；没有持久 freshness 或 validation 模型。

## 信号审计

| 信号来源 | 能证明 | 不能证明 | 可归因到单个技能 | 隐私与存储成本 |
|---|---|---|---|---|
| `Skill.createdAt` / `updatedAt`，`lib/db.ts:78-90`、`lib/skills.ts:42-73` | 技能何时创建；何时保存、改名、改正文、改 host、换来源或启停 | 何时使用、站点何时变化、内容是否仍有效 | 是，但只归因“这行被修改” | 已存在，无新增数据 |
| `matchSkillsForUrl` / `matchSkillsForTask`，`lib/skills.ts:14-39` | 某个已启用技能按 host 或 prompt 文本规则进入候选集合 | 模型是否看到摘要、是否读取、是否采用、是否帮助任务 | 是，匹配规则返回 `Skill.id`；当前未记录匹配事件 | 若持久化会增加任务与站点关联；当前不持久化 |
| 成功 `fs read /skills/<path>`，`lib/agent-tools.ts:143-150` | 当前任务成功读过该路径 | 模型是否采用内容、技能是否正确、页面动作是否依赖它 | 路径层面可以；tracker 不保存稳定 `Skill.id` | 当前只在内存保存路径，任务结束即丢弃 |
| 读取后的通用工具成功，`lib/agent-tools.ts:531-567` | 某个工具调用返回了结果；成功会清零连续异常计数 | 成功由哪个技能带来；`ok:false` 的可恢复结果也会走成功分支，因此不能当作技能有效 | 否。工具事件没有 skill-to-call 关系 | 事件和 tool run 已存在；建立关联会新增业务语义 |
| 连续两次抛错后的 nudge，`lib/agent-tools.ts:547-558` | 当前任务此前读过至少一条技能，随后累计了两次连续抛错，并向模型提示过一次 | 任一技能导致失败；`ok:false` 可恢复失败不会计入；提示后技能是否被复核 | 否。提示列出全部已读路径，没有因果关系 | 当前只改错误文本和任务内布尔值；未持久化提示状态 |
| Agent 通过 `fs write` 改写技能，`lib/fs-tool.ts:125-138`、`lib/skills.ts:42-66` | Agent 保存了一版新内容，`updatedAt` 随之变化 | 新内容已在真实站点验证；旧内容究竟哪里失效 | 是，成功写入落到一条技能记录 | 已存在；正文可能含站点流程，但规则禁止密钥和个人数据 |
| `task.completed` / `task.failed` / `task.cancelled`，`entrypoints/offscreen/main.ts:221-232` | 任务终态 | 任务是否依赖某条技能；完成是否代表每一步正确 | 否。任务事件没有技能关联 | 已存在；不得为归因保存 prompt、页面正文或思维链 |
| 用户显式确认“此技能有效/失效” | 若交互绑定稳定 `Skill.id`，能证明用户在该时点作出该判断 | 对所有 host、后续站点版本或其他任务持续有效 | 可以，但该信号目前不存在 | 需新增本地字段和交互；不需要保存 prompt、页面内容或个人资料 |

现有信号能描述“创建、修改、匹配、读取、任务遇到异常”。它们不能描述技能是否被采用，也不能把通用工具结果归因给某条技能。

## 状态语义

- **读取**：`fs read` 成功返回 `/skills/<path>`。读取是一次可观察的文件访问，不代表采用或有效。
- **修改**：技能记录成功保存或启停，`updatedAt` 更新。它表示本地内容或开关变化，不表示站点没有变化。
- **可能失效**：任务读过一条或多条技能后，连续两次工具调用抛错，系统发出一次 nudge。该状态属于任务级提示，不能落成单技能红标。
- **验证**：只有用户针对稳定技能 ID 的明确确认，或未来可证明某个结果依赖该技能的直接证据，才可称为验证。当前两类证据都不存在。
- **无可信状态**：没有用户确认时，UI 最多陈述来源、启用状态和最后编辑时间。空值不能解释为“未验证”“可靠”或“过期”。

任务成功、工具成功、读取后无报错都不升级为验证。多个技能同时被读取时，任何任务级结果都不能平均分配给这些技能。

## 方案比较

| 方案 | 用户价值 | 真实性 | 数据变更 | UI 成本 | 测试成本 | 误导风险 | 可逆性 | 判定 |
|---|---|---|---|---|---|---|---|---|
| **只展示已有事实**：来源、启用状态、最后编辑于 | 用户能判断谁创建了技能及本地多久没改；不能判断站点知识是否过期 | 高，字段已有明确语义 | 无 schema、索引或迁移 | 低，一行次要文案 | 低，覆盖相对时间和旧记录即可 | 中。用户可能把“近期编辑”读成“近期验证” | 高，删文案即可 | 不解决目标问题 |
| **最小使用遥测**：`lastReadAt`、`lastFailureHintAt` 或计数 | 能说明最近读过、某任务出现过失效提示 | `lastReadAt` 真实；`lastFailureHintAt` 对单技能只表达同任务共现，计数仍无因果 | 可选非索引字段无需 Dexie version；计数要事务，时间戳可直接覆盖 | 中，需要无状态、读取、提示、禁用和旧数据文案 | 中，需并发、时间边界、多技能和隐私测试 | 高。列表上的“提示”容易被理解成技能导致失败 | 中，字段可停写但历史含义长期存在 | 不应持久化单技能警告 |
| **显式验证流程**：用户针对技能确认有效或需要复核 | 信号最接近用户想知道的状态 | 用户确认本身真实；确认可能选错技能，也会随站点变化过时 | 需要本地状态、时间和来源；具体字段尚未获证据支持 | 高，需要任务后归因、技能选择、撤销和错误反馈 | 高，需跨 provider 任务上下文、误确认和恢复测试 | 中。若文案省略“用户确认”，仍会被当成系统验证 | 中，可删除入口，但已存判断需要迁移或清理 | 唯一可信候选，但缺少需求和交互证据 |

以下推导全部拒绝：

1. **从任务成功自动推导有效**：任务可能没有采用技能，也可能带着错误完成。
2. **给多个已读技能平均扣分**：共现不构成归因，平均分会制造精度。
3. **读取后任意失败即标红**：网络、权限、登录、页面状态和其他工具都能失败；当前 nudge 甚至只统计抛错，不统计 `ok:false`。

## 推荐

**No-Go。当前不实现站点技能新鲜度反馈。**

“最后编辑”没有回答新鲜度，“最后读取”没有回答有效性，任务级 nudge 又不能安全归因到技能。把这些字段压成列表状态会让用户把活动记录误读成质量判断。显式用户确认能产生可信信号，但项目没有用户需求证据、任务后交互样例或误确认数据，无法证明它的价值高于打扰和状态成本。

重新评估需要三类新证据：

1. 受控可用性测试证明用户需要在技能列表中判断是否复核，并能区分“读过”“用户确认”和“系统验证”。
2. 明确的确认入口能绑定稳定 `Skill.id`，多技能任务不要求用户猜哪条技能生效。
3. 失败归因夹具证明任何自动提示都不会把网络、权限、登录或其他技能问题写到单技能状态。

G4 不激活。

## 数据模型

本结论不授权持久状态。

| 决策面 | 决定 |
|---|---|
| schema | **无变化**。`Skill` 保持现有字段，不增加 `lastReadAt`、`lastFailureHintAt`、`validatedAt`、分数或计数 |
| 索引 / 迁移 | **无变化**。不新增索引，不提高 Dexie version；旧数据库没有默认 freshness 状态 |
| API / event | **无变化**。仍通过现有 `fs` 读写技能，固定六工具不变，不新增顶层工具、技能事件或 UI API |
| 并发 | **无变化**。没有新写入，也没有裸读、改、写；未来若引入计数必须用 Dexie 事务，时间戳只允许直接覆盖 |
| 缓存 / 认证 | **无变化**。不增加缓存、远端服务或认证策略 |
| 保留 / 删除 | **无变化**。技能继续独立于会话保留，删除技能即删除整行；会话删除不产生或清理 freshness 数据 |
| 导出 | **无变化**。会话导出不增加技能活动或确认记录 |
| 隐私 | 不保存 prompt、页面正文、工具输入输出、个人资料、凭证、原始思维链或站点内容副本 |
| 错误语义 | **无变化**。Dexie、`fs` 和任务错误沿用当前显式失败；不得在写入失败后显示新状态或假成功 |

## UI 状态

生产 UI 保持现有技能名称、host、描述、启停和删除。下表是静态文案审计，不是获准实施的界面。

| 场景 | 当前证据 | 360px 候选主状态 | 结论 |
|---|---|---|---|
| 新建 | `createdAt === updatedAt` 仅在尚未启停或重写时成立 | `刚刚创建` | 可陈述本地事实，不表示有效 |
| 近期编辑 | `updatedAt` | `3 天前编辑` | 可陈述本地事实，但不足以单独上线 |
| 只读过 | 任务内路径，任务后丢失 | 不显示；禁止补成“最近使用” | 无持久证据 |
| 出现失效提示 | 任务级 `hinted`，可能列出多个路径 | 不显示单技能警告 | 无法归因 |
| 禁用 | `enabled === false` | `已停用` | 已有可信状态，继续用中性色和现有开关 |
| 无可信状态 | 没有确认或可归因证据 | 不显示“可信”“已验证”“正常” | 空白不是质量结论 |
| 旧版本数据 | 只有当前 `Skill` 字段 | 与现有列表一致 | 不显示“从未使用”或“待验证” |

若未来证据支持显式确认，窄屏信息层级只能是一行主状态和按需展开的来源说明，例如 `用户于 3 天前确认`，不能写 `已验证`。颜色只给“用户报告需要复核”这类可操作警告；时间、读取和编辑状态使用中性色。

未来 UI 验收必须包含：所有开关和详情命中区至少 40px；键盘可打开详情、确认和撤销；screen reader 读出状态主体、确认来源与时间；360px 无横向溢出；状态变化复用 `fx-*`，并在 `prefers-reduced-motion` 下静止；技能状态只从 Dexie 单一事实源重建，不从 Svelte 本地状态恢复。

## 测试与验收

本 spike 不改测试。若重新评估，先用行为测试证明语义，再实现 UI：

- `scripts/test-skills.ts`：新字段的旧数据默认值、确认更新、撤销、删除、启停不冒充确认；只有索引变化才写 Dexie migration 测试。
- `scripts/test-agent-tools.ts`：只有成功 `fs read` 才能记读取；`ok:false` 与抛错分开；多个已读技能的失败不能写成任一技能的失败；审计持久化错误不产生 freshness 状态。
- 侧边栏 smoke：新建、近期编辑、禁用、旧数据、无可信状态和显式确认文案；360px、键盘、screen reader 名称、reduced motion。
- 隐私断言：数据库和导出中不出现 prompt、页面正文、个人资料、工具输入输出或 raw chain-of-thought 哨兵。
- 时间边界：未来相对时间使用注入时钟覆盖 59 秒、60 秒、24 小时和时钟回拨，不测试具体格式实现。

可量化试验指标不使用点击率：用户是否正确识别需要复核的技能、把“读取”误解为“验证”的比例、单技能错误归因数，以及每任务新增 Dexie 写入数。

## Go/No-Go

当前门槛结果：

| 门槛 | 通过标准 | 当前证据 | 结果 |
|---|---|---|---|
| 文案理解 | 10 名目标用户中至少 8 名能区分“读过”“用户确认”“系统验证”；最多 1 名把最后读取时间理解为可靠度 | 未测试 | 未通过 |
| 用户价值 | 至少 8 名用户能在 5 个带明确证据的技能卡片中选对 4 个需复核项，并说明状态依据 | 未测试 | 未通过 |
| 错误归因 | 单技能、多技能、网络、权限、登录和任务取消共 12 个夹具中错误归因数为 0 | 当前数据流无法归因 | 未通过 |
| 写放大 | 无显式确认时每任务新增写入为 0；每次确认最多 1 次事务写入 | 未设计确认流程 | 未通过 |
| 隐私 | 所有哨兵泄漏断言为 0 | 当前方案不写数据 | 仅现状通过 |

任一门槛未通过都维持 No-Go。上线后若错误归因不为 0、用户把状态当成系统质量保证，或每次技能读取都产生持久写入，应删除入口并停止写入；不得把文案改模糊来掩盖数据问题。

若未来全部门槛通过，实施仍需单独确认字段语义。候选改动边界是 `lib/db.ts`、`lib/skills.ts`、`lib/fs-tool.ts` 或明确的用户确认写入点、`entrypoints/sidepanel/Composer.svelte`、`lib/sidepanel-i18n.ts`、上述三类测试和 ADR 0015 的后续修订。不得新增顶层工具、远端遥测、后台定时器、轮询、质量分数或自动禁用。

最小发布顺序固定为：数据语义测试与隐私测试 → 事务写入和错误语义 → 窄侧边栏状态 → 真实 sidepanel smoke。评审只接受事实命名、零错误归因、有限写放大和可恢复的 Dexie 失败。
